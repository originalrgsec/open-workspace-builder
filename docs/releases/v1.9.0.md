# v1.9.0 — Release Pipeline Modernization + Consolidation (OWB-S118)

**Released:** 2026-04-11
**Sprint:** 23
**Tests:** 1,854 (+40 new S118 helper script tests, from 1,814)
**Theme:** Adopt GitHub Releases as the canonical release distribution
surface, ship a project SBOM as a Release asset on every tag push, and
recover from the ghost-release chain (v1.6.0/1.7.0/1.8.0 were tagged and
vault-closed but silently never reached PyPI due to a preexisting test-job
bug in the release workflow). v1.9.0 consolidates four sprints of work
into one cut and prevents this class of silent publish failure from
recurring.

## Consolidation Notice

**v1.9.0 is a consolidation release.** Between v1.5.0 (the last version
that actually reached PyPI) and v1.9.0, three minor versions were tagged
in the repository, documented in the vault, and declared shipped, but
never uploaded to PyPI. The `test` job in `.github/workflows/release.yml`
had been failing silently since Sprint 20 due to a missing `[sbom]`
extra on the wheel install, causing
`ModuleNotFoundError: No module named 'cyclonedx'` at pytest collection
time. The failure stopped `test`, which stopped `publish`, which meant no
PyPI upload. The tag push and the CHANGELOG entry gave the operator
every signal that the release had shipped, so the sprint-close checklist
(which did not verify the Release object or PyPI state) let each cycle
close without a flag.

PyPI latest was stuck at `1.5.0` for four sprints. Users who ran
`pip install open-workspace-builder` during that window have been on
v1.5.0 and have not seen any of the SBOM work from Sprints 20–22.
v1.9.0 ships all of that work to PyPI for the first time, together with
the Sprint 23 release pipeline fixes and the SEC-003 cryptography patch.

The ghost-release chain was discovered during the Sprint 23 RC rehearsal
for OWB-S118. The rehearsal forced the release pipeline to run
end-to-end for the first time in four sprints, and the first `rc1` tag
push failed in exactly the same `test` job that had been failing since
Sprint 20. Three iterations of the rehearsal (`v1.9.0rc1`, `v1.9.0rc2`,
`v1.9.0rc3`) were required to surface, fix, and validate all three
bugs. The full discovery sequence is documented in the Sprint 23 session
log at `Obsidian/projects/Open Source/open-workspace-builder/sessions/2026-04-11-sprint-23-rc-rehearsal.md`.

**Follow-up stories filed:**

- **OWB-S120** — fix `_REPO_ROOT` in `src/open_workspace_builder/sbom/_example.py`
  to survive an installed-wheel layout (the second preexisting bug surfaced
  during rehearsal).
- **OWB-S121** — vault errata for the v1.6.0/1.7.0/1.8.0 release manifests
  that claim PyPI publish happened when it did not.

**Consolidated feature inventory** (what a user upgrading from v1.5.0
receives, beyond the Sprint 23 changes below): the full S107 SBOM work
across three slices. See the release manifests for
[v1.6.0](v1.6.0.md), [v1.7.0](v1.7.0.md), and [v1.8.0](v1.8.0.md) for
the detailed per-sprint breakdowns; each of those tags is a real
revision in git and the manifests accurately describe what the tag
contained. The only fiction in those manifests is the "Released"
heading, which OWB-S121 will correct with an errata block.

## Code

| Module | Change |
|--------|--------|
| `.github/workflows/release.yml` | **Major rework.** The `test` job is replaced with a `smoke` job that installs the wheel into a clean environment with the `[sbom]` extra and verifies the CLI entry point (`owb --help`, `owb --version`). A new `github_release` job runs after `publish` succeeds: derives the version from the tag (stripping the `v` prefix), detects PEP 440 pre-release segments via regex (`(a\|b\|rc\|dev\|post)[0-9]+$`), extracts the CHANGELOG section matching the version via `scripts/extract_changelog.py`, installs `cyclonedx-python-lib` in the host environment, generates a project SBOM via `scripts/generate_sbom.py`, and creates a GitHub Release via `gh release create` with the wheel, sdist, and SBOM as assets. Pre-release tags produce a prerelease-flagged Release. The `publish` job `needs:` is updated from `test` to `smoke`. |
| `scripts/extract_changelog.py` | **New** — Keep-a-Changelog section parser. Reads `CHANGELOG.md`, finds the `## [version]` header matching the argument (exact match required, no `[Unreleased]` fallback), extracts the body until the next section header, trims leading and trailing blank lines, writes to stdout. Fails loud with a non-zero exit code on missing sections, whitespace-only sections, and malformed arguments. The release workflow treats a failure here as a hard stop so an empty or mismatched Release body cannot reach GitHub. |
| `scripts/generate_sbom.py` | **New** — project SBOM generator. Creates an isolated venv via `python -m venv`, installs the built wheel into it (only the wheel, no supplementary tooling), enumerates the installed distributions by running a small `importlib.metadata` snippet inside the venv and capturing JSON from stdout, then constructs a CycloneDX 1.6 BOM via `cyclonedx-python-lib` in the host environment. OWB is placed in `metadata.component` as `ComponentType.APPLICATION` with a `pkg:pypi/open-workspace-builder@<version>` purl; every other distribution is placed in `components` as `ComponentType.LIBRARY` with PEP 503 canonical names in its purl. Venv bootstrap packages (`pip`, `setuptools`, `wheel`, `distribute`, `pkg-resources`) are filtered by canonical name. The output is serialized via `JsonV1Dot6(bom).output_as_string(indent=2)` and validated to contain `bomFormat: "CycloneDX"` and the OWB subject in `metadata.component` before writing. |
| `tests/conftest.py` | New `load_script(name)` helper that loads a file from `scripts/` via `importlib.util` and returns the module. Used by both new test files so the importlib loader is not duplicated across test modules. |
| `src/open_workspace_builder/` | No changes. The S118 work is entirely release-pipeline-adjacent. The package itself does not grow new code. |

## Infrastructure

| Change | Notes |
|--------|-------|
| Release workflow shape | Four jobs: `build → smoke → publish → github_release`. Previously three jobs (`build → test → publish`) where `test` was silently broken. |
| Publish job `needs:` | Changed from `test` to `smoke`. Behavior is unchanged except that `smoke` now completes successfully. |
| New GitHub Actions permissions | The `github_release` job gets job-level `contents: write` to create the Release object. Top-level permissions (`id-token: write`, `contents: read`) are unchanged. |
| Host environment dependencies | The `github_release` job installs `cyclonedx-python-lib>=9.0,<11` directly via `pip` rather than installing the wheel with the `[sbom]` extra. Minimizes the host environment footprint and keeps OWB's own install surface separate from the BOM construction tooling. |

## Security

- **OWB-SEC-003** — `cryptography` bumped from `46.0.6` to `46.0.7`
  (CVE-2026-39892, Dependabot alert #16). Medium-severity buffer
  overflow in `Hash.update()` on non-contiguous buffers. In-context
  risk is low because OWB does not route user bytes into crypto APIs,
  but the patch is applied proactively.
- **Supply-chain quarantine override.** The `cryptography` 46.0.7
  release was published 2026-04-08, which placed it inside the 7-day
  post-release quarantine per
  `Obsidian/code/supply-chain-protection.md`. The natural quarantine
  window cleared 2026-04-15. The package owner authorized a four-day
  early override on 2026-04-11 to bundle SEC-003 into v1.9.0 rather
  than slipping the patch to a v1.9.1 cut. The override is recorded
  in:
    - `uv.toml` header comment (the canonical pin advancement log)
    - this release manifest (downstream-visible)
    - Sprint 23 session log (audit trail)
  Only `cryptography` was upgraded — `uv lock --upgrade-package
  cryptography` keeps every other package at its prior lock entry.
  Future pin advances should follow the same package-scoped pattern
  unless a global advance is genuinely warranted.
- **Transitive scope.** `cryptography` is a transitive dependency of
  `pyjwt[crypto]` (via `litellm` in the `[security]` extra) and
  `google-auth` (via `google-api-python-client` in the `[sheets]`
  extra). Users running `pip install open-workspace-builder` without
  any extras do not have `cryptography` installed at all and are not
  exposed to the CVE. Users with `[security]` or `[sheets]` get the
  patched version automatically once the lock advance is published in
  v1.9.0.

## RC Rehearsal Log

Three RC tags were pushed during Sprint 23 to validate the
`github_release` workflow end-to-end before the real v1.9.0 cut. Each
exposed a different preexisting bug:

| Tag | Outcome | Finding |
|-----|---------|---------|
| `v1.9.0rc1` | Failed in `test` | Missing `[sbom]` extra caused `ModuleNotFoundError: cyclonedx`. First visible manifestation of the ghost-release root cause. |
| `v1.9.0rc2` | Failed in `test` | With `[dev,sbom]` extras installed, the full suite ran but `tests/sbom/test_example_fixture.py` failed because `_REPO_ROOT = Path(__file__).resolve().parents[3]` resolves to `/opt/hostedtoolcache/Python/3.12.13/` when running against an installed wheel. Filed as OWB-S120. |
| `v1.9.0rc3` | **All four jobs green.** Build → smoke → publish → github_release. `cyclonedx-python-lib` SBOM correct (subject `open-workspace-builder 1.9.0rc3` application, purl `pkg:pypi/open-workspace-builder@1.9.0rc3`, components `[PyYAML, click]`, specVersion 1.6). Pre-release correctly flagged. PyPI pre-release published (`1.9.0rc3` permanently reserved). GitHub Release created with wheel + sdist + SBOM assets. |

After validation, the rc3 tag, the rc3 GitHub Release, and the scratch
branch (`sprint-23-release-rehearsal`) were all deleted. `1.9.0rc3`
remains on PyPI as an append-only artifact of the sanctioned rehearsal
procedure per AD-17.
