# v1.18.2 — Cross-Account CI Hardening (Sprint 36)

Released: 2026-05-02

## Summary

CI/CD chore release. No behavior change to OWB itself or the bundled content. Adopts the supply-chain hardening that AEL has shipped for several sprints by pinning every third-party Action across all four workflows to a 40-char commit SHA, and adds a `cancel-in-progress` concurrency block to `ci.yml` so superseded runs free up runner-minutes during force-pushes. Drives the cross-project tech-debt sprint TDB-10 / TD-CROSS-010 to closure.

This release also carries the v1.18.1-style operator-clean policy update (commit `26c239d`) ahead of the SHA-pin work — a +37 line section on the silent-failure mode for `@<sha>` references on cross-repo private→private reusable workflow calls, with the owner-controlled annotated-tag recovery pattern. Origin: AEL-S025 silent Slack-notify failures (2026-05-01). Public-repo reusable workflows and third-party Actions are unaffected and continue to follow the raw-SHA rule documented in §6.2.

Branch protection on `main` is applied via `gh api -X PUT` after the v1.18.2 release-workflow run completes — see *Branch protection* below.

## Branch protection (post-release)

After the release workflow completes and tags `v1.18.2`, branch protection is applied via:

```
gh auth switch -u originalrgsec
gh api -X PUT repos/originalrgsec/open-workspace-builder/branches/main/protection \
  -F required_status_checks.strict=true \
  -F 'required_status_checks.contexts[]=test (3.12)' \
  -F 'required_status_checks.contexts[]=test (3.13)' \
  -F 'required_status_checks.contexts[]=sast' \
  -F 'required_status_checks.contexts[]=dep-scan' \
  -F enforce_admins=true \
  -F required_pull_request_reviews.dismiss_stale_reviews=false \
  -F required_pull_request_reviews.required_approving_review_count=0 \
  -F restrictions= \
  -F allow_force_pushes=false \
  -F allow_deletions=false
```

Pre-state snapshot is captured to the TDB-10 session log before the call so the change is reversible.

## Upgrade Notes

- No runtime, API, or config changes. Existing `workspace.yaml` files and the installed wheel behave identically to v1.18.1.
- The bundled `supply-chain-protection.md` policy now documents the owner-controlled tag pin pattern. Existing workspaces are not auto-updated; consumers can pull the refresh via `owb update-content` or by copying the file.
- Action-pin rotations: when a pinned Action is rotated to a new SHA in a future sprint, update the trailing comment with the new tag/branch and the rotation date. Source PR or branch may be named in the trailing comment for richer audit trail.
